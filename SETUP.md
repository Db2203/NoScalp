# Setup — Aurora DSQL + Vercel

This walks through provisioning Aurora DSQL, running the app against it, and deploying to Vercel with OIDC (no long-lived keys).

## 1. Provision an Aurora DSQL cluster

Console → **Aurora DSQL** → *Create cluster*.

- **Single-region** is enough to run everything. For the multi-region demo (two endpoints that prove cross-region consistency), create a **multi-region** cluster — you'll get two regional endpoints (e.g. `us-east-1` + `us-east-2`) and a witness region.
- Note each endpoint host: `<cluster-id>.dsql.<region>.on.aws`.

DSQL has a recurring **free tier** (100k DPU + 1 GB/month), so a demo costs effectively nothing.

## 2. IAM permissions

The principal you connect as needs `dsql:DbConnectAdmin` (admin user) on the cluster. Minimal policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "dsql:DbConnectAdmin",
    "Resource": "arn:aws:dsql:*:<account-id>:cluster/*"
  }]
}
```

## 3. Run locally against DSQL

In `.env.local`:

```bash
NOSCALP_DB_MODE=dsql
DSQL_ENDPOINT_A=<cluster-id>.dsql.us-east-1.on.aws
DSQL_ENDPOINT_B=<cluster-id>.dsql.us-east-2.on.aws   # omit for single-region
DSQL_REGION_A=us-east-1
DSQL_REGION_B=us-east-2
NOSCALP_IDENTITY_SECRET=<random string>

# local credentials (the standard AWS chain)
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
AWS_REGION=us-east-1
```

Then:

```bash
npm run db:migrate    # creates tables + async indexes
npm run db:seed       # seeds the demo PS5 drop
npm run test:engine   # optional: prove the guarantees
npm run dev
```

The app authenticates with short-lived IAM tokens generated by `@aws-sdk/dsql-signer` (see `lib/db/pools.ts`); the token is used as the Postgres password over TLS.

## 4. Deploy to Vercel with OIDC (recommended)

Avoid putting AWS keys in Vercel. Use OIDC federation:

1. Vercel project → **Settings → Security → enable OIDC**. Note your issuer URL.
2. AWS IAM → **Identity providers → Add provider → OpenID Connect**, using Vercel's issuer and audience.
3. Create an IAM role trusting that provider (scope the trust to your Vercel team/project), attached to the policy in step 2.
4. In Vercel env vars set the role to assume (e.g. `AWS_ROLE_ARN`) plus `DSQL_ENDPOINT_A/B`, `DSQL_REGION_A/B`, `NOSCALP_IDENTITY_SECRET`. The AWS SDK's provider chain picks up the Vercel OIDC token automatically.

> All DB route handlers are pinned to the Node.js runtime (`export const runtime = "nodejs"`) because `pg` and the signer don't run on Edge.

## 5. Migrate the deployed DB

Run `npm run db:migrate && npm run db:seed` once against the cluster (locally with the `DSQL_*` env vars pointed at it, or from any machine with the IAM permissions).

## Notes / gotchas

- **`CREATE INDEX ASYNC`** returns before the index finishes building. The migration waits a few seconds; for a big cluster, give it a moment before seeding.
- **100 new connections/sec** is the real serverless ceiling — we use small, reused pools and batch inserts (the bot flood is ~5 transactions, not 5,000).
- **No foreign keys / triggers / SERIAL** — by design; relations are enforced in app code and all keys are UUIDs.
